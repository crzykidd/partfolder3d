# On-disk formats — sidecar schema & item naming

Authoritative spec for the **portable YAML sidecar** and the **title → on-disk name**
derivation. Implemented in **Phase 2** (storage/sidecar/item core). The sidecar is the
canonical, portable mirror of an item's metadata (PRD §1, §3.2): it must survive DB loss
and move cleanly between instances, so it stores **no instance-specific surrogate IDs**
(no DB `id`, no `user_id`) — identity travels via the stable `key`, everything else by
natural keys (tags by name, files by name + hash).

---

## 1. Sidecar file

- **Location/name:** `<itemname>-<key>/<itemname>-<key>.yml` (one per item, in its dir).
- **Encoding:** UTF-8 YAML. The app **regenerates** the file on write (a fresh header
  comment is emitted each time; do not rely on hand-added YAML comments surviving).
- **Reader is lenient:** unknown keys are ignored (forward-compat); missing optional keys
  default to null/empty.

### Example (schema_version 1)

```yaml
# PartFolder 3D item sidecar — portable metadata mirror. Regenerated by the app.
schema_version: 1

key: a7f3k9                        # stable, never changes — the item's real identity
title: "Ladybug Keychain"          # human title, verbatim (display uses this)
slug: ladybug-keychain-a7f3k9      # sanitized; matches the dir name and the URL

description: |
  Articulated ladybug keychain. Prints without supports.

source:
  url: https://www.printables.com/model/12345
  site: printables.com
  license: CC-BY-4.0

creator:                           # descriptive only — no local user_id (not portable)
  name: Jane Maker
  profile_url: https://www.printables.com/@janemaker
  source_site: printables.com
  is_original: false               # true = designed on the originating instance

tags: [keychain, articulated, toy] # flat canonical names

default_image: images/cover.png
images:
  - { path: images/cover.png,  source: scraped,  order: 0 }
  - { path: images/photo1.jpg, source: uploaded, order: 1 }

files:                             # paths relative to the item dir
  - { path: ladybug.3mf, role: model, size: 184320, sha256: 9f86d08…, mtime: 2026-06-20T14:03:11Z }
  - { path: project.zip, role: zip,   size: 952000, sha256: 2c26b46…, mtime: 2026-06-20T14:03:12Z }

created_at: 2026-06-21T09:15:00Z
updated_at: 2026-06-27T18:40:00Z
```

### Field semantics

| Key | Notes |
|---|---|
| `schema_version` | **Integer, starts at 1.** Bump **only** on breaking changes. Additive keys (e.g. Phase 7 `print_records`) do **not** bump it. |
| `key` | The 6–8 char base32 short hash. Portable identity; never changes (PRD §3.2/§18). |
| `title` | Verbatim human title (the display string). |
| `slug` | `<sanitized-title>-<key>` (see §2). Mirrors the dir name and the URL slug. |
| `source.{url,site,license}` | All optional. |
| `creator` | All fields optional/descriptive. `is_original: true` marks an original from the originating instance. **On import to another instance it maps to an external (named) Creator — it does NOT auto-bind to the importing user.** The local `Creator.user_id` link is set only when a user marks their *own* upload as self-designed (PRD §4). |
| `tags` | **Flat list of canonical tag names.** Category/namespace lives in the instance Tag vocabulary and is re-derived on import (not stored per-item here). |
| `default_image` / `images[]` | Paths **relative** to the item dir. `source` ∈ `scraped`/`uploaded`. |
| `files[]` | Inventory. `path` relative; `role` ∈ `model`/`zip`/`image`/`render`/`gcode`/`photo`/`other` (PRD §4); `sha256` lowercase hex; `mtime` ISO-8601 UTC. |
| `created_at` / `updated_at` | ISO-8601 UTC. |

### Hashing & drift policy

- **Algorithm: SHA-256** (lowercase hex). Universal/portable; recognizable across tools.
- **Cheap-first drift check:** treat a file as *unchanged* when `size` **and** `mtime`
  match the sidecar. Only **re-compute SHA-256 when size or mtime changed** (or on an
  explicit integrity pass / per-item Rescan). This keeps full hashing off the hot scan
  path even for very large (10GB+) libraries.
- **Legit-edit vs. corruption:** when a model file's recomputed hash differs from the
  baseline, the reconcile scan uses the **mtime direction** to disambiguate — a *newer*
  mtime on a file that still parses is a legitimate in-place edit (adopt the new hash,
  re-render, no Issue), while an *unchanged/older* mtime, or a newer mtime on an
  unparseable file, is flagged as `corruption`. See
  [`features-overview.md`](features-overview.md#corruption-vs-legitimate-in-place-edits).

### Sidecar ⇄ DB conflict resolution

Out of scope for this doc — governed by the admin setting in PRD §8.2 / §18(4):

## 2. Title → on-disk name (sanitization)

The verbatim `title` is preserved for display. The filesystem **and** the URL use a single
sanitized form so they never diverge: `dir = <slug>-<key>`, `URL path = <slug>-<key>`.

**Algorithm (deriving the slug body, before the `-<key>` suffix):**

1. Unicode-normalize (NFKD) → **transliterate to ASCII** (e.g. `é`→`e`).
2. Lowercase.
3. Replace every char outside `[a-z0-9]` with `-`.
4. Collapse runs of `-`; trim leading/trailing `-`.
5. If the result is empty (e.g. an all-CJK or all-emoji title), fall back to the base
   string **`item`**.
6. **Cap the slug body at 80 characters** (cut on a char boundary; re-trim trailing `-`).

Then the on-disk dir / URL is `<slug-body>-<key>`.

**Properties:**
- **Collisions are impossible by construction** — the unique `-<key>` suffix
  disambiguates identical titles (`ladybug-a7f3k9` vs `ladybug-b2k8m1`).
- The `-<key>` suffix also neutralizes Windows reserved device names (`con-a7f3k9` is not
  reserved) and step 4's trimming removes the leading-dot / trailing-dot / trailing-space
  filesystem traps.
- **Rename** (the only thing that moves a dir) re-derives the slug body from the new title;
  `<key>` is invariant, so links/downloads/API refs survive (PRD §3.2, §8.5).
